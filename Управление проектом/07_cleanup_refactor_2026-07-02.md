# Cleanup / refactoring сайта

Дата: 2026-07-02  
Стартовый опубликованный коммит: `f575bd0 Polish shop cards and social preview`  
Цель: убрать технические наслоения без визуального изменения текущего сайта.

## Зачем

Сайт уже близок к финальному состоянию, но CSS и часть JS накоплены итерационными слоями. Это нормально для быстрой дизайн-доводки, но дальше такая структура повышает риск: новая маленькая правка может неожиданно перебить старое правило или сломать light/night/mobile.

## Как это называется

- technical debt / техдолг;
- cleanup;
- refactoring;
- CSS cascade cleanup;
- visual regression hardening.

## Текущие факты

- `site/css/style.css`: 21 426 строк до первого cleanup-прохода.
- `!important`: около 4 494 вхождений.
- `CYCLE`-блоки: 111.
- Есть повторные слои для header, shop, footer, project overlay, night theme.
- После удаления footer-ссылки `О сайте` остался мертвый код privacy-модалки.

## Правило безопасности

Cleanup считается успешным только если выполняются все пункты:

1. Визуально сайт выглядит так же на ключевых экранах.
2. Консоль браузера без ошибок.
3. `git diff --check` чистый.
4. Нет изменений в контенте, которые не относятся к cleanup.
5. Старые незавершенные материалы в рабочем дереве не трогаются.
6. После публикации проверяется GitHub Pages с новым cache-bust.

## Первые безопасные шаги

1. Убрать мертвую privacy-модалку после удаления ссылки `О сайте`.
2. Убрать оставшиеся `.footer-privacy`-хвосты из CSS, но только после проверки футера.
3. Составить карту последних override-блоков `CYCLE 79-92`.
4. Схлопывать один модуль за раз:
   - shop;
   - project overlay;
   - header / language toggle;
   - night theme;
   - footer.
5. После каждого модуля делать отдельный коммит.

## Что пока не трогать

- Не менять сетку проектов вместе с cleanup.
- Не менять типографику вместе с cleanup.
- Не удалять trial-шрифты.
- Не трогать грязные рабочие материалы:
  - `materials/docs/projects_master.csv`;

  - `research/`, `output/`, `site/multipage/`, `site/font-test.html`, trial fonts и другие незавершенные файлы.

## Независимый аудит Dirac

Dirac подтвердил, что главный риск - не JS, а каскад `site/css/style.css`.

Самые рискованные зоны:

- `shop`: много поздних правил для `.shop-grid--reliefs`, `.shop-buy`, `.shop-price`;
- `project overlay`: несколько слоев правят layout, overflow и галерею;
- `header / lang / nav`: hard-lock правила для капсул, языка, burger и сердечка;
- `night theme`: слои `CYCLE 79-84` и более поздние overrides частично перекрывают друг друга;
- JS-классы нельзя переименовывать механически, потому что `main.js` читает состояния `site-night`, `project-inline-overlay`, `shop-story`, `cv-*` и другие.

Рекомендация Dirac:

- не дробить `style.css` первым шагом;
- не трогать поздние блоки `CYCLE 79-92` без baseline-скриншотов;
- сначала удалять только явно мертвый код;
- для каждого модуля делать отдельный cleanup-коммит;
- проверять desktop/mobile/light/night и открытые состояния.
