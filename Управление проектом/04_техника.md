# Техника

## Текущая база

Сайт статический: HTML, CSS, JavaScript, JSON.

Папка сайта: `site/`.

Текущие страницы:

- `index.html`;
- `about.html`;
- `shop.html`;
- `projects/project-1/index.html`.

Устаревшие файлы `projects.html`, `template_.html`, `js/script.js` и заглушка `projects/project-2/` удалены из живой структуры сайта. История сохранена в Git.

## Архитектура первой версии

Рекомендация:

- оставить статический сайт без тяжелой сборки;
- хранить контент в JSON;
- фото хранить по папкам проектов;
- публичные документы для скачивания хранить в `site/assets/docs/`;
- для важных проектов оставить физические HTML-страницы или шаблон;
- RU/EN хранить рядом в структуре данных;
- поддержать поле `videos`.

## v0-контентный скелет

`v0` уже собран как рабочий контентный скелет:

- статический сайт с реальной навигацией;
- главная с примерно 15 записями, где проблемные проекты можно скрывать из витрины;
- данные из CSV/CV;
- реальные фото для готовых проектов;
- временные изображения для недостающих проектов;
- проектные страницы пока не считаются публикационно готовыми;
- без финального дизайна, но с рабочей адаптивной структурой.

Техническое правило: временный контент должен быть отделен от опубликованного, чтобы его нельзя было случайно принять за финал.

Текущая раскладка `v0` - рабочая. Она уже должна показывать направление: композиционная витрина вместо ровной сетки одинаковых карточек.

Статус: первая `v0`-витрина собрана через `projects.json`.

- 15 записей в данных, 14 видимых карточек на главной;
- у проектов `Маяк` и `Души не чаю - Ярославль` включен прототип story-card: фото листаются внутри карточки на главной;
- страница `Все мои друзья` временно не считается опубликованной: в ней ошибочно стоят фото `Маяка`;
- 14 видимых карточек на главной, 1 запись скрыта до настоящих фото;
- битых путей в `projects.json` нет;
- старая заглушка `project-2` убрана из страницы проекта и fallback-данных;
- добавлены реальные папки фото для первых 10 проектов;
- 10 карточек уже получили облегченные web-обложки из реальных фото;
- `Маяк` добавлен как отдельный проект: Екатеринбург, 2024, с 6 web-фото для story-card;
- `Души не чаю - Ярославль` получил 6 web-фото для story-card;
- `Души не чаю` разделен на три карточки: Ярославль, Иваново, Вологда;
- в исходных материалах есть локальные видео (`.mov`, `.mp4`), структуру видео нужно поддержать в данных.

### Как развивать v0 на текущей базе

Текущая главная уже строит карточки из `site/assets/data/projects.json`.

Дальше можно:

- уточнить список 10-15 записей в `projects.json`;
- для проектов без фото использовать временные изображения из корня сайта или отдельную временную обложку;
- не считать `project-1` полноценной страницей, пока в ней стоят ошибочные фото;
- для остальных проектов временно отключать переход, пока не утверждена страница проекта;
- не создавать 15 HTML-страниц вручную до согласования состава проектов;
- добавить поля статуса: `draft`, `needs-images`, `v0`, `published`.

Граница: `v0` не должен маскироваться под финальную публикацию. Если карточка временная, это должно быть видно в данных и в рабочем процессе.

### Минимальная v0-задача

## Скорость

Требования:

- сайт должен быстро загружаться;
- mobile не должен получать тяжелые оригиналы;
- изображения должны быть сжаты для веба;
- видео не должно блокировать загрузку страницы;
- анимации должны быть легкими и не мешать производительности;
- перед публикацией нужна проверка веса и скорости.

Текущий факт: исходная папка `materials/source-photos/` весит около 1.2 ГБ. Это рабочие оригиналы, не сайт-ассеты. В папку сайта переносим только подготовленные web-копии.

В новых папках уже есть 4 локальных видео: три по серии Самоката и одно по проекту `Июль`. Для `v0` их можно хранить как источник, но не подключать к главной до решения формата.

## Анимация интерфейса

Анимация нужна как часть характера сайта, но должна быть простой и изящной.

Подходящие зоны:

- hover-состояния кнопок и ссылок;
- плавные переходы карточек;
- аккуратные раскрытия/переключения;
- смена языка;
- переходы внутри галерей;
- микро-движение, которое поддерживает типографику и проекты.

Ограничения:

- не делать тяжелые декоративные эффекты;
- не мешать просмотру работ;
- не ухудшать mobile;
- учитывать `prefers-reduced-motion`;
- motion должен быть системным, а не случайным набором эффектов.

## Композиция галереи

Нужно предусмотреть, что проектная галерея может иметь разные паттерны раскладки:

- 2 карточки в ряд на mobile;
- 1 крупная карточка после пары маленьких;
- чередование рядов;
- разные пропорции изображений;
- редакционная раскладка вместо равномерной сетки.

Технически это можно решать через данные проекта:

- `layoutSize`: `small`, `medium`, `large`;
- `layoutGroup`;
- `featured`;
- `sortOrder`.

Для `v0` уже используется черновой паттерн: обычные карточки, широкие карточки и крупные карточки. Для следующего дизайн-этапа нужно отдельно спроектировать финальные паттерны на реальных фото.

Дополнительно начат паттерн `story-card`: избранный проект может иметь набор изображений прямо в карточке главной, счетчик, кнопки листания, метаданные, короткое описание и вторичную ссылку на полную страницу. Сейчас паттерн нужно сравнить на двух разных проектах: `Маяк` и `Души не чаю - Ярославль`.

Мобильное правило: не сводить галерею к одной колонке. Даже на телефоне должен оставаться ритм: пары небольших карточек, затем крупная карточка, затем другое чередование.

## Обновление сайта

Рекомендация для первой версии:

1. Мастер-список работ вести в `materials/docs/projects_master.csv`.
2. Опубликованные и v0-проекты хранить в `site/assets/data/projects.json`.
3. Фото хранить в папках проектов.
4. Обновления делать редко и проверять вручную.

CMS или админку рассматривать позже, если обновления станут частыми.

## Git

Git включен для проекта.

В Git храним:

- сайт `site/`;
- управленческие документы;
- CSV/PDF-источники из `materials/docs/`;
- публичные web-копии документов в `site/assets/docs/`;
- подготовленные web-ready изображения внутри сайта.

Не храним в Git:

- `.DS_Store`;
- тяжелые оригиналы `materials/source-photos/`;
- локальные референсы `materials/references/`;
- временные файлы, сборки и зависимости.

Правило: перед каждым коммитом проверять, что в историю не попали исходники весом из рабочих фото-папок.
